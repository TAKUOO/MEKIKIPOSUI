## 1. 企業・プロジェクト概要

### 1-1. 会社ビジョン

全国に存在する中小の質屋・買取店が「鑑定士」というレイヤーで一致団結し、
フランチャイズではない**連合体**を形成することで、大手に対抗しうる集団を作っていく。

> **Claudeへ**：このビジョンは機能設計の根幹である。「個店の効率化」だけでなく「加盟店同士のネットワーク価値」を常に意識して提案すること。

### 1-2. プロダクト全体像

```
┌──────────────────────────────────────────────────────────────┐
│                        MEKIKI cloud                           │
│                                                               │
│  ① Community POS  ←──→  ② Community Hub                     │
│  （店舗業務）              （加盟店間連携）                     │
│         ↕                         ↕                          │
│  ④ monowallet     ←──→  ③ MEKIKI Academy                    │
│  （コンシューマー）         （教育コンテンツ）                   │
│                                                               │
│         ↑①④両方に搭載          ↑                            │
│  ⑤ MEKIKI BRAIN（生成AIベース査定エンジン・自社開発・API提供） │
└──────────────────────────────────────────────────────────────┘
```

| # | プロダクト名 | 利用者 | 概要 |
|---|------------|--------|------|
| ① | **Community POS** | 加盟店スタッフ・オーナー | 質屋・買取業務の一元管理 |
| ② | **Community Hub** | 加盟店 ↔ 本部 ↔ 加盟店同士 | コミュニケーション・情報共有 |
| ③ | **MEKIKI Academy** | 加盟店スタッフ・鑑定士 | 動画・音声による教育コンテンツ配信 |
| ④ | **monowallet** | コンシューマー（一般消費者） | 会員カード ＋ 動産価値確認アプリ |
| ⑤ | **MEKIKI BRAIN** | ①POS・④monowallet（API経由） | 自社開発の生成AIベース査定エンジン。品物情報・写真をもとに査定価格を算出。POSの買取・入質時と、monowalletの動産価値表示の両方に搭載 |

## 3. 業界用語・ドメイン定義

> ⚠️ Claudeはこのセクションの定義に厳密に従うこと

| 用語 | 定義 | 備考 |
|------|------|------|
| **入質**（にゅうしち） | 顧客が品物を担保にお金を借りる取引（質預かり受付） | 画面上「入質入力」として表示 |
| **出質**（でしち） | 顧客が返済して品物を取り戻す取引 | 画面上「出質」として表示 |
| **流質**（りゅうしち） | 返済期日を過ぎて品物の所有権が店に移転すること | 画面上「流質処理」として表示 |
| **質物台帳**（しちもつだいちょう） | 質預かり中の品物を管理する帳簿 | 質屋営業法の法定帳簿 |
| **古物台帳**（こぶつだいちょう） | 買取品の仕入記録帳簿 | 古物営業法の法定帳簿 |
| **貸付金**（かしつけきん） | 質入れ時に顧客に渡す融資額 | 「残元金」とも表示される |
| **残元金**（ざんもときん） | 現時点での未返済の貸付元本 | 質物台帳・棚卸で表示 |
| **流質期限**（りゅうしちきげん） | この日を過ぎると質流れになる返済期限 | 画面では赤字で警告表示 |
| **棚番**（たなばん） | 品物の保管場所コード（例：赤ケース、2-4-2など） | 棚卸画面で管理 |
| **質料**（しちりょう） | 質預かりに対する利息・手数料 | 法定上限あり（月5.0%が現行運用） |
| **流率**（りゅうりつ） | 流質率（質流れになった割合） | 顧客別に管理 |
| **回数** | 顧客の取引累計回数 | 顧客の取引頻度指標 |
| **買取**（かいとり） | 顧客から品物を直接購入する取引 | 所有権即時移転 |
| **仕入区分** | 品物の取得方法区分：流質・買取・仕入・調整 | ラベル発行時に選択 |
| **グループ** | 品目の大分類（例：貴金属地金、時計、バッグなど） | 買取入力時に選択 |
| **品目** | 品物の中分類（例：指輪、ネックレス、財布など） | 買取入力時に選択 |
| **程度** | 品物のコンディション（現行：数値入力） | 🔲 グレード制への変更検討中 |
| **点数** | 1取引で扱う品物の個数 | |
| **ポイント** | 買取額に連動した顧客還元ポイント（買取額の1%が現行） | |
| **担当CD** | 取引を担当したスタッフのコード | |
| **棚卸**（たなおろし） | 保管中の質物・在庫品の実数確認作業 | |
| **質物（籠）棚卸** | 流質期限が過ぎた・または保管中の質物の一覧管理 | 「籠」は質蔵の意 |
| **金券業務** | 商品券・ギフト券等の買取・販売業務 | メインメニューに独立 |
| **鑑定士** | 品物の真贋・価値を判断できる専門家。連合体の核 | |
| **加盟店** | MEKIKI cloudを導入している質屋・買取店 | |
| **本部** | 吉田悟率いるMEKIKI運営チーム | |
| **動産** | 不動産以外の財産。monowalletの管理対象 | |

#### P-001：取引明細入力（個人別）

**概要**：顧客を特定し、過去の取引履歴を確認した上で新規取引（質 or 買取）を選択する起点画面。

**確認済み表示項目**
- 日付（和暦対応）
- 顧客コード・顧客名（カナ）
- 性別・生年月日・年齢
- 通知設定
- 電話番号下4桁
- 回数（累計取引回数）
- 相利（累計利益率 %）
- 取引履歴一覧：日付 / 台帳No. / 品目 / 明細 / 点数 / 買取金額 / 獲得ポイント / 発行ポイント / 担当者名
- 合計買取金額
- 特記事項（「店長買い取り分」等のフリーテキスト）
- 住所

**確認済みアクション**
- 新規登録選択ダイアログ：「質 F1」「買取 F2」「取消」の3択

**MEKIKIでの改善点（現行からの変更）**
- ✅ 日付は西暦に統一（和暦表示廃止）
- ✅ 顧客検索は顧客コード・氏名・カナ検索に対応
- ✅ タブレット対応UI（現行はPCキーボード操作前提）

#### P-002：買取入力

**概要**：顧客から品物を買い取る際の明細入力画面。

**確認済み入力項目（1行 = 1品物）**
- グループ（品目大分類）
- 品目（品目中分類）
- メーカー
- 品名（自由記述）
- 程度
- 点数
- 買取額
- 摘要

**確認済み操作**
- 商品検索（F1）：マスタから商品を検索して入力
- 行追加（F4）：複数品物を1伝票で処理可能
- 前行複写（F6）：直前の行をコピー
- 商品修正（F10）：入力済み商品の修正
- 登録（F12）

**確認済み精算画面（P-005 買取精算）**
- 買取金額
- 点数
- 今回獲得ポイント（買取額の1%）
- 前回累計ポイント

- 担当者コード・氏名
- 次回累計ポイント
- 登録方法：「登録（伝票）」「登録（未発行）」「キャンセル」

**MEKIKIでの改善点**
- ✅ ポイント制度を継承（買取額の1%付与）
- ✅ バーコード / QRコード / RFID による商品識別（方式検討中）
- ✅ 商品画像の撮影・添付機能を追加
- ✅ MEKIKI BRAIN API連携による査定参考価格の表示
- 🔲 程度のグレード表記（S/A/B/C等）への変更検討

#### P-003：入質入力 / P-004：貸付登録

**概要**：顧客の品物を担保に融資する質預かり処理。

**確認済み入力項目（入質入力画面）**

**MEKIKIでの改善点**
- ✅ 利率5.0%（月・満月計算）を現行仕様として引き継ぐ
- ✅ 流質期限の自動計算（期間選択→自動算出）

## 4-5. MEKIKI BRAIN 連携仕様

> **Claudeへ**：MEKIKI BRAINはMEKIKI社が自社開発している生成AIベースの査定エンジンである。外部サービスではなく自社プロダクトの一部として扱うこと。

### 概要
- 品物の情報（カテゴリ・ブランド・型番・状態・写真等）を入力すると、AIが査定価格を算出するAPI
- POS側（買取・入質時のスタッフ向け査定支援）と monowallet側（コンシューマー向け動産価値表示）の両方に搭載

### POS側での利用シーン
| シーン | 使い方 |
|--------|--------|
| 買取受付 | 品物情報・写真を入力するとMEKIKI BRAINが買取参考価格を提示。スタッフの査定を補助 |
| 入質受付 | 査定額フィールドにMEKIKI BRAINの算出結果を参照・入力 |
| Community Hub | 他加盟店への買取相談時にBRAINの査定結果を添付 |

### monowallet側での利用シーン
| シーン | 使い方 |
|--------|--------|
| 動産価値表示 | 登録した動産の写真・情報をもとにBRAINがおおよその市場価値を算出・表示 |
| 事前査定依頼 | ユーザーが加盟店に査定依頼する際の参考価格として活用 |

### 連携方式
- ✅ API経由（RESTまたはGraphQL）
- 🔲 APIエンドポイント・認証方式・レートリミット等の仕様は別途MEKIKI BRAIN側と確認が必要

### 未定・検討中
- [ ] MEKIKI BRAIN APIの仕様書・エンドポイント共有（BRAIN開発チームから受領要）
- [ ] 査定結果の精度・免責事項の表示方針（特にmonowallet側）
- [ ] オフライン時のフォールバック動作（BRAINが応答しない場合の処理）

### WF-001：買取ワークフロー（新規顧客）

#### 全体フロー図

```
【店舗側】

STEP 1: 顧客来店・商品持ち込み
  └─ 新規顧客登録へ（WF-001-A）

STEP 2: 査定
  ├─ 商品写真を撮影
  ├─ MEKIKI BRAIN APIで価格予測
  └─ 予測結果の評価
        ├─ [予測成功] → STEP 3へ
        └─ [予測困難 / 精度不足] → 相談フロー（WF-001-B）へ
                                     └─ 相談結果を受けて STEP 3へ

STEP 3: 買取価格の提示
  └─ スタッフが顧客に口頭で提示

STEP 4: 顧客の判断
  ├─ [買取成立] → STEP 5へ
  └─ [買取不成立] → 査定見積もり保存フロー（WF-001-C）へ → 終了

STEP 5: 本人確認
  └─ 身分証明書を写真撮影・システムに取り込み

STEP 6: 買取明細入力
  └─ 商品詳細（グループ・品目・メーカー・品名・程度・買取額等）を入力

STEP 7: 電子サイン（買取同意書）
  └─ iPadの画面上で顧客がサイン

STEP 8: 金銭授受
  └─ 現金または振込

STEP 9: 買取明細書の発行
  ├─ [monowallet登録済み] → アプリに買取データ送信
  └─ [未登録] → 紙の明細書を印刷して渡す

STEP 10: 商品バーコード印刷・在庫登録
  └─ ラベルを商品に貼付 → 在庫へ
```

## 7. monowallet（コンシューマー向けネイティブアプリ）

> **Claudeへ**：monowalletはPOSと双方向に連携する。POS側での買取・査定の結果がリアルタイムにアプリへ反映されることが重要な設計要件である。

| 機能ID | 機能名 | 概要 | ステータス |
|--------|--------|------|-----------|
| M-001 | 動産登録 | 品物情報・写真を登録して管理 | 🔲 未定 |
| M-002 | 動産価値表示 | MEKIKI BRAIN APIで登録動産のおおよその市場価値を算出・表示 | ✅ 仕様確定 |
| M-003 | **デジタル会員カード** | 加盟店でのQRコードスキャンによる顧客識別。既存顧客の呼び出しに使用 | ✅ 仕様確定 |
| M-004 | **売却履歴** | 加盟店での買取成立時に商品画像・買取金額が自動で記録される | ✅ 仕様確定 |
| M-005 | **査定見積もり履歴** | 買取不成立時に商品画像・査定見積もり金額が保存・閲覧できる | ✅ 仕様確定（新規） |
| M-006 | **ポイント残高・履歴** | 買取ポイントの残高確認・付与履歴の閲覧（加盟店がポイント制を採用している場合） | ✅ 仕様確定 |
| M-007 | 近隣加盟店検索 | 位置情報を使った加盟店マップ | 🔲 未定 |

| M-009 | プッシュ通知 | 相場変動・キャンペーン等の通知 | 🔲 未定 |

## 8. データモデル概要

### 主要エンティティ

```
顧客（Customer）
  ├── 本人確認情報（IdentityVerification）
  │     ├── 証明書種別・No・有効期限
  │     └── 画像ファイル（クラウドストレージ参照）
  ├── 身体特徴（PhysicalFeature）※防犯用
  └── 取引統計（CustomerStats）※自動集計

取引（Transaction）
  ├── 買取（Purchase）
  │     ├── 買取明細（PurchaseDetail） × N品
  │     └── 精算（PurchaseSettlement）
  │           └── ポイント付与
  └── 入質（Pawn）
        ├── 質物明細（PawnDetail） × N品
        └── 貸付登録（PawnLoan）
              ├── 貸付金・利率・期間・流質期限
              └── 棚番（保管場所）

質物（PawnItem） ← 保管中の質物
  ├── 残元金
  ├── 流質期限
  ├── ステータス（保管中 / 出質 / 流質）
  └── 利明細（InterestDetail）

在庫品（InventoryItem）
  ├── 仕入区分（流質 / 買取 / 仕入 / 調整）
  ├── 商品CD（自動採番）
  ├── グループ / 品目 / メーカー / 品名
  ├── 程度 / 点数
  ├── 原価（仕入価格）
  ├── 売価
  ├── 棚番
  └── ステータス（在庫中 / 販売済み）

売上（Sale）
  ├── 売上明細（SaleDetail） × N品
  ├── 得意先（業者 or 一般）
  └── 入金内訳（現金 / 振込 / 掛）

法定帳簿
  ├── 質物台帳（PawnLedger） ← 入質ごとに自動生成
  └── 古物台帳（AntiqueLedger） ← 買取ごとに自動生成

担当者（Staff）
  └── 担当コード / 氏名

【外部API連携】
MEKIKI BRAIN（自社開発・生成AIベース査定エンジン）
  ├── 入力：グループ / 品目 / メーカー / 品名 / 状態 / 画像
  ├── 出力：査定参考価格 / 信頼スコア / 根拠情報
  └── 呼び出し元：買取入力 / 入質入力（POS側）/ 動産価値表示（monowallet側）
```

## 9. 非機能要件

### 9-1. 法的要件（絶対遵守）

| 法令 | 要件 |
|------|------|
| 古物営業法 | 買取記録（古物台帳）を3年間保存。売主の本人確認必須 |
| 質屋営業法 | 質料の上限規制遵守（現行5.0%/月・満月計算）。質物台帳の保存 |
| 個人情報保護法 | 顧客情報・身分証画像の暗号化保存・適切な管理 |
| 消費税法 | 外税/内税/非課税の区分管理。10%対応 |

### 9-2. セキュリティ

- 顧客の個人情報・身分証データの暗号化保存
- ロールベースアクセス制御：本部管理者 / 加盟店オーナー / スタッフ / 鑑定士 / コンシューマー
- 全操作ログの保存（誰が・何を・いつ）
- 加盟店間のデータ分離（A店の顧客情報がB店から閲覧不可）

### 9-3. 対応デバイス

- ✅ POS：タブレット優先（PC兼用も考慮した設計）
- ✅ monowallet：iOS / Android ネイティブアプリ
- ✅ 商品識別：バーコード / QRコード / RFID のいずれか（方式は検討中）
- ✅ 本人確認：マイナンバーカードICチップ読取に対応

### 9-4. データ移行

- ✅ 現行「楽質」システムからのデータ移行を実施する
- ✅ 日付フォーマットは西暦に統一（和暦→西暦の変換処理が必要）
- 🔲 移行対象範囲の確定：顧客マスタのみか / 台帳履歴（質物・古物）も含むか
- ⚠️ 和暦台帳No.（H18/01/28等）の西暦変換ロジックは別途設計が必要
